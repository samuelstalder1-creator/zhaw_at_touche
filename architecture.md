# Architecture

This document describes the two model families used in this project: **fine-tuned classifiers** and **delta models**. Both families solve the same binary classification problem — given a search-engine response (and optionally its query), decide whether the response contains advertising language. They differ fundamentally in what is trained, what the model sees at inference time, and what computational assumptions they make.

The per-setup configuration reference is in `setup.md`. The result table and confusion matrices are in `results.md`. The research question analysis is in `research_question.md`.

---

## Background: the neutral rewrite

Both families optionally exploit a **neutral rewrite** — a version of the response with any promotional language removed, generated by a language model (Gemini or Qwen). If a response contains an ad, the delta between the original and the rewrite should expose it. If a response is clean, the rewrite should be nearly identical.

The neutral rewrite is the bridge between the two families: fine-tuned classifiers can read it as part of their input text, while delta models treat it as a reference embedding to subtract from the response embedding.

## Fine-tuned Classifiers

### Concept

A pre-trained transformer encoder (RoBERTa, Longformer, DeBERTa, etc.) is fine-tuned end-to-end on the classification task. The model receives a text sequence as input, and a binary classification head on top of the `[CLS]` token produces a logit. Both the backbone weights and the classification head are updated during training.

The key property is that **the encoder learns task-specific representations**: after fine-tuning, the internal activations are no longer general-purpose sentence embeddings but are shaped specifically by which token patterns distinguish ad responses from clean ones.

### What is trained

- All transformer encoder weights (all layers, all attention heads)
- A two-class linear head (`hidden_size → 2`)

The optimizer (AdamW) runs with layer-wise weight decay, optional gradient clipping, and a warmup-then-decay learning rate schedule. The class imbalance (~70% negative, 30% positive) is compensated via an optional `positive_class_weight_scale` multiplier applied to the cross-entropy loss.

Training is configured through `train_model/<setup>.json` and executed via the `classifier` or `cross_encoder` trainer type in `modeling.py`.

### Input formats

The text sequence fed to the tokenizer varies by setup:

| Format | What the model sees | Setups |
|---|---|---|
| `response_only` | `[CLS] response [SEP]` | setup115 |
| `query_response` | `[CLS] query [SEP] response [SEP]` | setup6, setup12, setup8–10 |
| `query_neutral_response` | `[CLS] query [SEP] neutral [SEP] response [SEP]` | setup7, setup7-qwen |
| `query_dual_neutral_response` | `[CLS] query [SEP] gemini_neutral [SEP] qwen_neutral [SEP] response [SEP]` | setup116 |
| `cross_encoder` | `[CLS] response [SEP] neutral [SEP]` | setup105_1 |

For formats that include the neutral, the tokenizer sees the response and neutral in the same forward pass. The encoder can attend across both texts at every layer — it is not comparing fixed vectors, it is attending to the specific tokens that differ between them.

The `cross_encoder` format is a special case: it is the most expressive neutral-aware design because it allows the model to identify the exact token-level differences between response and neutral rather than comparing compressed summaries. It uses the same `classifier` training loop but with a different input format.

Long-context setups (query + neutral + response) use Longformer-base-4096 because RoBERTa's 512-token limit is too short to fit all three segments. Longformer's sliding-window attention makes it practical for these inputs while the global `[CLS]` token still aggregates the full sequence for classification.

### Inference

The trained model loads from `models/<setup>/` via `AutoModelForSequenceClassification`. Input text is tokenized, forwarded through the model, and the `[CLS]` logit is converted to a probability via softmax. Predictions above 0.5 are labelled as positive (ad). No separate threshold calibration is needed because the classification head already produces calibrated logits.

## Delta Models

### Concept

Instead of fine-tuning the encoder, the encoder stays **completely frozen**. The idea is that a strong general-purpose sentence encoder (e.g., `all-mpnet-base-v2`) already represents semantic content accurately enough that the difference between the response embedding and the neutral embedding directly encodes what the neutral rewrite removed — i.e., the advertising signal.

Formally: let `r = encode(response)` and `n = encode(neutral)`. The delta is `δ = r − n`. If the response is an ad, `δ` points in the direction of advertising language in embedding space. If the response is clean, `δ ≈ 0`.

A **logistic regression** (with a standard scaler) is then trained on top of the frozen delta vectors. Only the logistic regression weights and the scaler are learned — the encoder is never updated. This separates representation learning (done offline by the pre-trained encoder) from decision boundary learning (done by the LR on top of fixed features).

### What is trained

- A `StandardScaler` fitted to the training feature matrix (zero-mean, unit variance per dimension)
- A `LogisticRegression` (scikit-learn, L2 regularization by default) fitted on the scaled feature matrix
- A decision threshold calibrated on the validation set to maximize a target metric (default: macro F1)

The encoder (`all-mpnet-base-v2`) is never updated. No GPU training loop, no backpropagation through the backbone. Embeddings are computed once in batched GPU forward passes and cached; the logistic regression is then fitted on CPU.

Training is configured through `train_model/<setup>.json` and executed via the embedding LR trainer types in `embedding_lr_classifier.py`.

### Feature variants

Different setups construct different feature vectors from the same frozen embeddings:

| Setup | Feature vector | Dim | What it encodes |
|---|---|---:|---|
| setup103 | `r − n_gemini` | 768 | Gemini residual — what Gemini removed |
| setup119 | `r − n_qwen` | 768 | Qwen residual — what Qwen removed |
| setup104 | `[r \| n_gemini \| r − n_gemini]` | 2304 | Residual + absolute positions |
| setup113 | `[r − n_gemini \| r − n_qwen]` | 1536 | Two independent residuals |
| setup114 | `[r \| n_gemini \| n_qwen \| r − n_gemini \| r − n_qwen]` | 3840 | Full dual-provider stack |
| setup117 | `[q \| r − n_gemini]` | 1536 | Gemini residual + query topic context |
| setup118 | `[q \| r − n_gemini \| r − n_qwen]` | 2304 | Dual residual + query topic context |

Where `r` = response embedding, `n_gemini`/`n_qwen` = neutral embeddings, `q` = query embedding.

The simplest variant (setup103, `r − n`) is also the strongest single-file delta model at 0.9913 Macro F1. Adding absolute embedding positions (setup104) shifts precision/recall trade-offs but does not improve overall F1. Adding a second neutral (setup113) introduces some noise from the weaker Qwen neutral, dropping to 0.9857. Adding the query (setup117, setup118) hurts badly because the 768-dim query embedding introduces noise that competes with the residual signal in the LR weight space.

### Scalar baselines (setup100–102, setup110–111)

An earlier family collapsed the same embeddings to scalar distances (cosine similarity between pairs of texts) before applying a threshold or logistic regression. These fail because a single scalar discards the directional information in the delta vector — two responses can drift the same cosine angle from their neutral while one is promotional and the other is not. Setup100–102 score 0.35–0.45 F1; setup110–111 score ~0.57 F1 using six scalars from two providers. The 768-dim delta (setup103) outperforms all of them at 0.9913 by preserving direction.

### Inference

The saved bundle (`models/<setup>/embedding_lr_classifier.pkl`) contains the fitted `StandardScaler`, the fitted `LogisticRegression`, and the calibrated threshold. At inference time, the encoder embeds the configured text fields, the feature vector is assembled, scaled, and passed through the LR to produce a probability. Predictions above the calibrated threshold are labelled as positive (ad).

## Comparison

| | Fine-tuned classifier | Delta model |
|---|---|---|
| **Encoder** | Updated during training | Frozen, general-purpose |
| **What is trained** | All backbone weights + classification head | Logistic regression + scaler only |
| **Training cost** | GPU, hours | GPU for embedding (once), CPU for LR |
| **Neutral required** | Optional (in-prompt or cross-encoder) | Required (as reference embedding) |
| **Neutral use** | Attended to jointly in the same forward pass | Subtracted from response embedding |
| **Best Macro F1** | 0.9987 (setup115, response only) | 0.9913 (setup103, Gemini residual) |
| **Errors (FP+FN)** | 7 (setup115) | 45 (setup104) / 46 (setup103) |
| **Query helps?** | No — response alone beats all query-aware models | No — query actively hurts (0.9913 → 0.7224) |
| **Second neutral helps?** | Marginally in prompted classifiers | No — dual residual trails single Gemini |

The fine-tuned classifier is stronger in every matched comparison. The delta model is a strong lightweight alternative that reaches near-ceiling performance with no backbone training — but the gap of ~0.007 Macro F1 (46 vs 7 errors) shows that end-to-end adaptation of the encoder still matters.
